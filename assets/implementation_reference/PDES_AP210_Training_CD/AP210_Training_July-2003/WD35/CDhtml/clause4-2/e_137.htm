4.2.137 Component_terminal

NOTE: The text description for this Application Object has not been fully updated to agree with the EXPRESS.
A Component_terminal is a type of Component_feature. A Component_terminal is one of an Assembly_module_component_terminal, a Bare_die_component_terminal, a Component_termination_passage_join_terminal, a Conductive_interconnect_element_terminal, an Interconnect_component_join_terminal, an Interconnect_module_component_terminal, a Land_join_terminal, a Non_functional_land_join_terminal, a Packaged_component_join_terminal, a Printed_component_join_terminal, or a Via_terminal. A Component_terminal is the defined conductive shape for conductors from other components to access electrical functionality of the Assembly_component.

The data associated with this application object are the following:

  • connected_node
  • global_swappable
  • local_swappable
  • node_implementation
  • swap_code

4.2.137.1 connected_node

The connected_node is an inverse relationship that specifies the Physical_connectivity_definition that specifies the Component_terminal as its associated_terminals. There shall be no more than one Physical_connectivity_definition for a Component_terminal. The connected_node need not be specified for a particular Component_terminal.

4.2.137.2 global_swappable

The global_swapable specifies a boolean value that indicates if the terminal can be exchanged within associated functional units of the same type.

4.2.137.3 local_swappable

The local_swappable specifies a boolean value that indicates if the terminal can be exchanged within the associated functional unit.

4.2.137.4 node_implementation

The node_implementation is an inverse relationship that specifies the Physical_network that specifies the Component_terminal as its connectivity_requirement_element. There shall be no more than one Physical_network for a Component_terminal. The node_implementation need not be specified for a particular Component_terminal.

4.2.137.5 swap-code

The swap_code specifies the string that is the type of exchange that is valid.

Informal constraints:

IP1: Each Component_terminal that is referenced by a Physical_connectivity_definition as an associated_terminal shall not also be in the design view of the implementation of that Physical_connectivity_definition.

IP2: The node_implementation and the connected_node shall not both exist.
